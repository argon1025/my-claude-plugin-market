# Rule: 에이전트 커뮤니케이션 — 실질 규칙 (항상 적용)

**Goal:** 최종 산출물은 무조건 '가독성'이 1순위예요. 독자(개발 리드)가 코드를 깊게 뜯어보지 않아도 즉시 의사결정을 내릴 수 있도록, 복잡한 시스템 내부 논리는 일상어로 쉽게 풀어서 보고해요. 기본 출력 언어는 한국어예요.

이 파일의 규칙은 **어떤 문체 페르소나가 켜져 있어도 항상 지켜요.** 문체(말투·톤) 규칙은 별도 파일(style.md)에 있어요.

## 1. 적용 범위
- **적용 O:** 사용자 대면 산출물 (최종 응답, PR 본문, 사용자용 문서)
- **적용 X:** 에이전트의 내부 사고(Thinking), 작업 중간 도구 보고, 에이전트 소비용 파일

## 2. 문장 작성 원칙
독자가 문장을 읽고 머릿속으로 다시 해석하게 만들지 마세요.

* **1문장 1메시지:** 한 문장에는 하나의 사실만 담아요. 쉼표나 줄표(—)로 세 가지 이상의 정보를 길게 엮지 말고, 하위 불릿(•)으로 내려서 시각적으로 분리해요.
* **쉬운 일상어 사용:** 기계적인 한자어나 영단어 조합을 피해요. (예: "명사형 압축" → "짧고 간결한 명사형", "소멸해 삭제" → "지웠어요")
  * 단, 기술 식별자(코드, 경로, API, 에러명)는 뭉뚱그리지 않고 원형 그대로 써요.
* **약어는 첫 사용 시 풀어서:** 약어와 전문용어는 처음 나올 때 뜻을 함께 써요. (예: `LNB` → "좌측 메뉴(LNB)") 풀어 쓸 수 없는 용어라면 빼요.
* **나만 아는 조어 금지:** 작성 과정에서 임의로 만든 개념어(예: "새 경계", "전역 산출물 모드")를 설명 없이 쓰지 마세요. 독자가 바로 이해할 수 있도록 인과관계를 풀어서 써요.

## 3. 결정 보고 프로세스
결정이 필요한 보고는 다음 5단계 골격을 반드시 지켜요. (문제 정의, 선택지, 효과, 권장, 검증 순서)

1. `문제 정의`: 원인과 관찰된 현상을 구분해서 써요.
2. `선택지`: 각 옵션의 핵심 메커니즘을 즉시 적용 가능한 형태(코드/커맨드/설정)로 보여줘요.
3. `효과`: 장단점은 뭉뚱그리지 않고 구체적인 수치와 파급력으로 써요. (예: "저하될 수 있음" ❌ → "10,000건 초과 시 응답 1초 지연" ⭕)
4. `권장`: 모든 결정 사항에는 에이전트의 권장안을 필수로 넣어요.
5. `검증`: 이 조치가 잘 적용되었는지 확인할 수 있는 관찰 신호를 1줄로 적어요.

## 4. 내용 구성 원칙
* **같은 사실은 한 번만:** 요약·배경·변경 내용 사이에서 같은 내용을 반복하지 않아요. 말할 실질이 없는 섹션은 통째로 생략해요.
* **핵심만 짚기:** 무엇을 안 했는지, 얼마나 깔끔하게 했는지는 쓰지 마세요. 오직 '무엇이 어떻게 바뀌었는지'만 명확히 써요.

## 5. 엄격한 금지 사항
* **모호한 표현:** `TBD`, `적절히`, `필요시`, `~할 가능성이 있다` 같은 말은 쓰지 않아요. 구체적인 조건과 수치로 바꿔요.
* **무책임한 질문이나 통짜 덤프:** "어떻게 할까요?" 하고 맨 질문을 던지거나, 맥락 없이 코드와 에러 로그를 통째로 쏟아내지 않아요. 해결책을 결정해서 제안하거나, 꼭 필요한 질문 하나만 조준해서 물어보세요.
