# Rule: 에이전트 커뮤니케이션 — 실질 규칙 (항상 적용)

**Goal:** 최종 산출물은 무조건 '가독성'이 1순위예요. 독자(개발 리드)가 코드를 깊게 뜯어보지 않아도 즉시 의사결정을 내릴 수 있도록, 복잡한 시스템 내부 논리는 일상어로 쉽게 풀어서 보고해요. 기본 출력 언어는 한국어예요.

이 파일의 규칙은 **어떤 문체 페르소나가 켜져 있어도 항상 지켜요.** 문장의 완결성과 문단 구조는 실질이라서 페르소나에 양보하지 않아요. 어투(어미·말투) 규칙만 별도 파일(style.md)에 있고, 그쪽만 페르소나에 양보해요.

## 1. 적용 범위
- **적용 O:** 사용자 대면 산출물 (최종 응답, PR 본문, 사용자용 문서)
- **적용 X:** 에이전트의 내부 사고(Thinking), 작업 중간 도구 보고, 에이전트 소비용 파일

## 2. 문장 작성 원칙
독자가 문장을 읽고 머릿속으로 다시 해석하게 만들지 마세요.

* **완결문 필수:** 산출물의 모든 문장은 서술어로 끝내요. 전보문·명사문("외부 서비스 ~10곳 미대응", "실패 = redirectToLogin")을 쓰지 마세요. 어미 스타일(해요체/평서체)은 활성 페르소나를 따르되, 문장 완결성 자체는 어떤 페르소나에서도 지켜요.
* **기호 논리 금지:** `=`, `+`, `→` 같은 기호로 논리를 표현하지 마세요. (예: "실패 = 무조건 redirectToLogin" ❌ → "발급에 실패하면 무조건 `redirectToLogin()`을 호출해요" ⭕)
* **1문장 1메시지:** 한 문장에는 하나의 사실만 담아요. 쉼표나 줄표(—)로 세 가지 이상의 정보를 길게 엮지 말고, 하위 불릿(•)으로 내려서 시각적으로 분리해요.
* **쉬운 일상어 사용:** 기계적인 한자어나 영단어 조합을 피하고, 한자어 명사 압축은 동사로 풀어요. (예: "미대응" → "아직 대응하지 못했어요", "무변경" → "바꾸지 않아요", "신설" → "새로 만들어요")
  * 단, 기술 식별자(코드, 경로, API, 에러명)는 뭉뚱그리지 않고 원형 그대로 써요.
* **약어는 첫 사용 시 풀어서:** 약어와 전문용어는 처음 나올 때 뜻을 함께 써요. (예: `LNB` → "좌측 메뉴(LNB)") 풀어 쓸 수 없는 용어라면 빼요.
* **나만 아는 조어 금지:** 작성 과정에서 임의로 만든 개념어(예: "새 경계", "전역 산출물 모드")를 설명 없이 쓰지 마세요. 독자가 바로 이해할 수 있도록 인과관계를 풀어서 써요.
* **포인터 참조 금지:** "(결정 7)", "plan.md 144행 참고"처럼 내용을 다른 곳으로 떠넘기지 마세요. 그 내용을 한 줄로 요약해서 문장 안에 넣고, 위치 표기는 보조로만 병기해요.

## 3. 문단 구성 골격
문단은 라벨 나열이 아니라 문장의 흐름으로 읽히게 써요.

* **두괄식 선언문:** 문단의 첫 문장은 핵심 주장 한 줄이에요. (예: "문제는 인증 방식이에요.") 근거와 수치는 뒤 문장에 둬요.
* **인과 연결어로 흐름 연결:** 문장 사이 논리는 "그래서", "하지만", "왜냐하면" 같은 연결어로 이어요. 라벨 나열로 논리 전개를 대체하지 마세요.
* **용어 즉석 재정의:** 새로 등장하는 개념이나 정책 결정은 바로 다음 문장에서 풀어요. (예: "여기서 '없앤다'라는 건 발급 로직을 서버로 옮긴다는 뜻이에요.")
* **수치는 그대로:** 수치는 단순하게 그대로 보여줘요. (예: "대상 10곳") 서사적 대비 구조로 포장하지 않아요.
* **문단은 짧게, 빈 줄로 구분:** 한 문단은 4문장 이내로 써요. 화제가 바뀌면 반드시 빈 줄로 끊고, 내용 덩어리가 3개 이상이면 소제목이나 라벨로 섹션을 나눠요. 문단 구분 없이 텍스트를 한 덩어리로 이어 쓰지 마세요.

## 4. 결정 보고 프로세스
결정이 필요한 보고는 다음 5단계 골격을 반드시 지켜요. (문제 정의, 선택지, 효과, 권장, 검증 순서)

1. `문제 정의`: 원인과 관찰된 현상을 구분해서 써요.
2. `선택지`: 각 옵션의 핵심 메커니즘을 즉시 적용 가능한 형태(코드/커맨드/설정)로 보여줘요.
3. `효과`: 장단점은 뭉뚱그리지 않고 구체적인 수치와 파급력으로 써요. (예: "저하될 수 있음" ❌ → "10,000건 초과 시 응답 1초 지연" ⭕)
4. `권장`: 모든 결정 사항에는 에이전트의 권장안을 필수로 넣어요.
5. `검증`: 이 조치가 잘 적용되었는지 확인할 수 있는 관찰 신호를 1줄로 적어요.

## 5. 내용 구성 원칙
* **같은 사실은 한 번만:** 요약·배경·변경 내용 사이에서 같은 내용을 반복하지 않아요. 말할 실질이 없는 섹션은 통째로 생략해요.
* **핵심만 짚기:** 무엇을 안 했는지, 얼마나 깔끔하게 했는지는 쓰지 마세요. 오직 '무엇이 어떻게 바뀌었는지'만 명확히 써요.

## 6. 엄격한 금지 사항
* **모호한 표현:** `TBD`, `적절히`, `필요시`, `~할 가능성이 있다` 같은 말은 쓰지 않아요. 구체적인 조건과 수치로 바꿔요.
* **무책임한 질문이나 통짜 덤프:** "어떻게 할까요?" 하고 맨 질문을 던지거나, 맥락 없이 코드와 에러 로그를 통째로 쏟아내지 않아요. 해결책을 결정해서 제안하거나, 꼭 필요한 질문 하나만 조준해서 물어보세요.
* **자작 라벨 골격 금지:** "무엇:", "왜 이 모양:", "진 대안:"처럼 즉석에서 만든 라벨로 보고 골격을 짜지 마세요. 라벨은 4장의 5단계 골격(문제 정의/선택지/효과/권장/검증)에 있는 것만 쓰고, 그 외 구획은 문장형 소제목으로 나눠요.

## 7. 적용 예시
같은 계획 브리핑을 아래처럼 고쳐 써요.

**나쁜 예 (전보문 압축):**

> 무엇: 영역별 `loadHandoffCode.ts` 신설. 실패 = 무조건 `redirectToLogin`.
> 왜 이 모양: 기존 복제 컨벤션 미러. 공용화 시 두 로더가 다른 위치 규칙 갖게 됨.
> 리스크 (plan.md 144행): 장애 시 90여 페이지 전부 로그인 튕김.

**좋은 예 (완결문 + 두괄식):**

> 이번 커밋은 10개 앱 영역마다 `loadHandoffCode.ts`를 새로 만들어요. 공용 모듈 1개가 아니라 10개를 복제하는 이유는, 기존 `loadIframeCredential.ts`가 이미 영역별 복제 방식을 쓰고 있어서예요. handoff만 공용화하면 두 로더가 서로 다른 위치 규칙을 갖게 돼요.
>
> 발급에 실패하면 페이지 구분 없이 무조건 `redirectToLogin()`을 호출해요. 그래서 발급 API에 장애가 나면 iframe 페이지 90여 곳이 전부 로그인 페이지로 이동해요(리스크는 plan.md 144행에도 적어뒀어요).
> - 검증: 배포 후 `/login` 리다이렉트율과 `content-iframe/issue-handoff-code` MLog 에러율을 확인해요.
