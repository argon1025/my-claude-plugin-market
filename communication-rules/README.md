# communication-rules

커뮤니케이션 규칙 플러그인이에요. 에이전트의 최종 산출물을 읽기 쉬운 보고 형식으로 만들어줘요. 별도 스킬은 없어요.

## 동작 방식

규칙을 성격에 따라 두 파일로 나눠서 주입해요.

- `rules/substance.md` — **실질 규칙.** 결정 보고 골격, 구체적 수치, 모호어 금지, 기술 식별자 원형 유지. 어떤 문체 페르소나가 켜져 있어도 항상 지켜요.
- `rules/style.md` — **문체 규칙.** 해요체 완결문, 이모지·군더더기·번역투 금지. 다른 문체 페르소나(예: caveman)가 켜져 있으면 최종 응답에서는 그 페르소나에 양보해요.

훅은 두 시점에 실행돼요.

- **SessionStart** (`hooks/session_start.sh`): 두 규칙 파일 전체를 세션 컨텍스트에 주입해요.
- **UserPromptSubmit** (`hooks/user_prompt_submit.sh`): 매 턴 실질 규칙만 2줄로 리마인드해요. 세션이 길어지거나 컨텍스트가 압축돼도 규칙이 사라지지 않게 하기 위해서예요. 문체 규칙은 리마인드에 넣지 않아서, caveman 같은 페르소나의 매 턴 리마인드와 나란히 떠도 충돌하지 않아요.

## 적용 범위

- 사용자 대면 산출물(최종 응답, PR 본문, 사용자용 문서 파일)의 작성 규칙만 담아요.
- 에이전트의 내부 사고(thinking)나 작업 중간 도구 보고는 규정하지 않아요.
- PR 본문과 사용자용 문서 파일에는 페르소나와 무관하게 문체 규칙까지 전체 적용해요.

## 적용 전 / 적용 후

같은 작업 결과를 보고할 때 이렇게 달라져요.

**적용 전**

> sessionKey 검증 로직 리팩토링 진행 후 배포 수행 필요. 인덱스 미적용 상태로 데이터 많을 시 성능 저하 가능성 존재하며, 필요시 캐시 도입 검토 예정.

**적용 후**

> 세션키 검증 로직을 정리했어요. 배포 전에 결정할 게 하나 있어요.
>
> - 문제 정의: `session` 테이블에 인덱스가 없어서, 지금은 검증 때마다 전체를 훑어요.
> - 효과: 세션이 10,000건을 넘으면 응답이 1초 이상 느려져요.
> - 권장: `session_key` 컬럼에 인덱스를 추가하고 배포해요.
> - 검증: 배포 후 세션 검증 API의 응답 시간이 100ms 아래로 유지되는지 확인해요.

## 규칙 요약

- 한 문장에는 하나의 사실만 담고, 병렬 정보는 하위 불릿으로 내려요. (실질)
- 기계적인 한자어·조어 대신 일상어로 쓰되, 기술 식별자(코드·경로·API·에러명)는 원형을 유지해요. (실질)
- 결정 보고는 문제 정의, 선택지, 효과, 권장, 검증의 5단계 골격을 지켜요. (실질)
- 모호한 표현(`적절히`, `필요시`)과 맨 질문·통짜 덤프를 금지해요. (실질)
- 해요체 완결문을 쓰고, 인사말·이모지·AI 관용구·번역투를 금지해요. (문체 — 페르소나에 양보)
