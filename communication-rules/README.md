# communication-rules

커뮤니케이션 규칙 플러그인이에요. 에이전트의 최종 산출물을 읽기 쉬운 보고 형식으로 만들어줘요. 별도 스킬은 없어요.

## 동작 방식

규칙을 성격에 따라 두 파일로 나눠서 주입해요. 경계는 '실질+문장 구성 vs 어투'예요.

- `rules/substance.md` — **실질 규칙.** 완결문·두괄식 문단 같은 문장 구성, 결정 보고 골격, 구체적 수치, 모호어 금지, 기술 식별자 원형 유지. 어떤 문체 페르소나가 켜져 있어도 항상 지켜요.
- `rules/style.md` — **어투 규칙.** 해요체, 이모지·군더더기·번역투 금지. 다른 문체 페르소나(예: caveman)가 켜져 있으면 최종 응답에서는 그 페르소나에 양보해요.

문장의 완결성과 문단 구조를 실질로 분류한 이유가 있어요. 가독성은 어미가 아니라 문장 구성에서 나오는데, 이걸 어투로 취급하면 간결체 페르소나가 켜졌을 때 장문 산출물까지 전보문 압축체로 나와요.

훅은 두 시점에 실행돼요.

- **SessionStart** (`hooks/session_start.sh`): 두 규칙 파일 전체를 세션 컨텍스트에 주입해요.
- **UserPromptSubmit** (`hooks/user_prompt_submit.sh`): 매 턴 실질 규칙만 2줄로 리마인드해요. 세션이 길어지거나 컨텍스트가 압축돼도 규칙이 사라지지 않게 하기 위해서예요. 어투 규칙은 리마인드에 넣지 않아서, caveman 같은 페르소나의 매 턴 리마인드와 나란히 떠도 충돌하지 않아요.

## 적용 범위

- 사용자 대면 산출물(최종 응답, PR 본문, 사용자용 문서 파일)의 작성 규칙만 담아요.
- 에이전트의 내부 사고(thinking)나 작업 중간 도구 보고는 규정하지 않아요.
- PR 본문과 사용자용 문서 파일에는 페르소나와 무관하게 어투 규칙까지 전체 적용해요.

## 적용 전 / 적용 후

같은 계획 브리핑을 보고할 때 이렇게 달라져요.

**적용 전 (전보문 압축)**

> 무엇: 영역별 `loadHandoffCode.ts` 신설. 실패 = 무조건 `redirectToLogin`.
> 왜 이 모양: 기존 복제 컨벤션 미러. 공용화 시 두 로더가 다른 위치 규칙 갖게 됨.
> 리스크 (plan.md 144행): 장애 시 90여 페이지 전부 로그인 튕김.

**적용 후 (완결문 + 두괄식)**

> 이번 커밋은 10개 앱 영역마다 `loadHandoffCode.ts`를 새로 만들어요. 공용 모듈 1개가 아니라 10개를 복제하는 이유는, 기존 `loadIframeCredential.ts`가 이미 영역별 복제 방식을 쓰고 있어서예요. handoff만 공용화하면 두 로더가 서로 다른 위치 규칙을 갖게 돼요.
>
> 발급에 실패하면 페이지 구분 없이 무조건 `redirectToLogin()`을 호출해요. 그래서 발급 API에 장애가 나면 iframe 페이지 90여 곳이 전부 로그인 페이지로 이동해요.
> - 검증: 배포 후 `/login` 리다이렉트율과 `content-iframe/issue-handoff-code` MLog 에러율을 확인해요.

## 규칙 요약

- 모든 문장은 서술어로 끝내고, 기호 논리(`=`, `+`, `→`)와 한자어 명사 압축("미대응", "무변경")을 금지해요. (실질)
- 문단 첫 문장은 핵심 주장 한 줄로 쓰고, 문장 사이는 인과 연결어로 이어요. (실질)
- 한 문장에는 하나의 사실만 담고, 병렬 정보는 하위 불릿으로 내려요. (실질)
- "(결정 7)" 같은 포인터 참조 대신 내용을 인라인으로 요약해요. (실질)
- 결정 보고는 문제 정의, 선택지, 효과, 권장, 검증의 5단계 골격을 지키고, 즉석에서 만든 라벨 골격("무엇:", "왜 이 모양:")을 금지해요. (실질)
- 모호한 표현(`적절히`, `필요시`)과 맨 질문·통짜 덤프를 금지해요. (실질)
- 한 문단은 4문장 이내로 짧게 쓰고, 화제가 바뀌면 빈 줄로 끊어요. (실질)
- 해요체를 쓰고, 인사말·이모지·AI 관용구·번역투를 금지해요. (어투 — 페르소나에 양보)
